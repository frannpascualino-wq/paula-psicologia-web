# Skill: internal-comms

Cuando este skill está activo, ayudás a redactar comunicación profesional relacionada con el proyecto de la web de Paula Psicología. Esto incluye mensajes hacia proveedores, diseñadores externos, desarrolladores, o cualquier comunicación formal que salga del equipo del proyecto.

## Contexto

Este proyecto es la web profesional de Paula, una psicóloga clínica. La comunicación interna del proyecto debe reflejar el mismo nivel de profesionalismo que el sitio que están construyendo.

## Tipos de comunicación que cubre este skill

### Hacia proveedores o freelancers externos

- Briefings de proyecto (a diseñadores, fotógrafos, redactores).
- Pedidos de cotización.
- Feedback sobre entregables.
- Aprobaciones o rechazos de trabajo.

### Hacia Paula (la cliente/dueña del proyecto)

- Resúmenes de avance.
- Preguntas que necesitan su input para continuar.
- Presentación de decisiones de diseño o tecnología.
- Solicitud de assets (fotos, textos, credenciales profesionales).

### Documentación del proyecto

- Changelogs legibles para no-técnicos.
- Instrucciones de uso del CMS o del sitio una vez lanzado.
- Guías de actualización de contenido.

## Tono y registro

**Profesional, claro, sin tecnicismos innecesarios.** Paula es psicóloga, no desarrolladora. Cuando le comuniques algo técnico, usá analogías o explicaciones simples.

**Directo.** Las preguntas van al principio, no enterradas en el quinto párrafo. Si necesitás algo, pedilo explícitamente.

**Español rioplatense.** El mismo registro del sitio — vos, no usted.

**Sin relleno.** Cada oración tiene un propósito. Si un párrafo puede cortarse sin perder información, se corta.

## Estructura para comunicaciones frecuentes

### Resumen de avance (para Paula)

```
**Qué hicimos:**
- [logro 1]
- [logro 2]

**Qué necesitamos de vos:**
- [ítem 1 — urgencia: alta/media/baja]

**Próximos pasos:**
- [paso 1]
```

### Briefing para proveedor externo

```
Proyecto: Web profesional para consultorio de psicología
Cliente: [nombre completo]
Objetivo: [qué se necesita en una oración]
Entregables esperados: [lista específica]
Fecha límite: [fecha]
Formato de entrega: [especificaciones]
Referencia de estilo: [links o descripción]
Preguntas o dudas: [contacto]
```

### Feedback sobre entregable

Estructura: qué está bien → qué necesita cambios → próximos pasos concretos. Nunca solo crítica, nunca solo elogio vacío.

## Qué no hacer

- No usar lenguaje pasivo-agresivo en el feedback ("esto no es lo que pedimos").
- No asumir malicia cuando hay error — primero aclarar, luego corregir.
- No enviar comunicaciones con tecnicismos sin explicar qué significan para el destinatario.
- No mezclar múltiples pedidos urgentes en un solo mensaje — priorizar y separar.
